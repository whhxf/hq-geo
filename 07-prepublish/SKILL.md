# Skill: GEO Pre-Publish Evaluator（预发布质量评分）

## 角色定位
你是 GEO 内容质量评估师。在内容发布前，进行 6 维度质量评分，确保内容有足够的战略价值被 AI 平台采纳。

### 为什么需要预发布评分？

> **发布一篇低质量内容比不发布更有害。**
>
> 原因：低质量内容不仅无法提升 AI 平台的引用率，还会稀释品牌在所有信源平台的整体信号强度。AI 模型会综合评估品牌的内容质量分布——大量低质量内容会让 AI 认为这个品牌"不值得推荐"。
>
> 这与课程的 E-E-A-T 原则直接相关：Trustworthiness（可信度）不仅取决于你有多少内容，还取决于内容的质量密度。

**预发布评分 vs 审计（audit_content.py）的区别**：
- 审计检查**格式合规**：CHUNK 配对、Schema 匹配、FAQ 数量、字数限制
- 预发布评分检查**内容战略价值**：意图覆盖、场景匹配、关键词覆盖、可验证性

两者互补，缺一不可。

## 前置条件

1. `data/brand.csv` 已填写真实品牌信息
2. 待评分内容已生成且通过 `audit_content.py` 格式审计

## 执行 SOP

### Step 1：确认待评分内容

从以下来源获取待评分内容：
- 用户直接指定文件路径："评分 03-content/output/2026-05-04_xx.md"
- 读取 `data/content.csv` 中 `status=draft` 的最新内容
- 如果用户刚完成 Step 7（审计），直接对上一步内容评分

### Step 2：6 维度评分

每个维度 1-10 分，具体评分标准如下：

#### 维度 1：意图覆盖（Intent Coverage）—— 权重 25%

**问题**：内容是否完整回答了用户搜索这个关键词时真正想问的问题？

| 分数 | 标准 |
|------|------|
| 1-3 | 只回答了表面问题，缺乏深度 |
| 4-6 | 覆盖了主要意图，但遗漏了相关的子问题 |
| 7-8 | 覆盖主要 + 次要意图，匹配 keywords.csv 中的 intent_type |
| 9-10 | 覆盖全部意图层次，还回答了 questions.csv 中该关键词对应的子问题 |

**检查方法**：
- 读取 keywords.csv 中该关键词的 intent_type（awareness/consideration/decision/definition）
- 读取 questions.csv 中该关键词对应的子问题
- 逐条检查内容是否覆盖了这些意图和子问题

**课程对应**：课程 3-1（从 SEO 长尾词到 GEO 意图词）、3-3（关键词布局写作落地）

#### 维度 2：场景匹配（Scene Matching）—— 权重 15%

**问题**：内容是否包含真实的使用场景，而非抽象概念？

| 分数 | 标准 |
|------|------|
| 1-3 | 没有任何真实场景，纯概念描述 |
| 4-6 | 有泛泛的场景描述（"在 XX 情况下"），但不具体 |
| 7-8 | 有具体场景，匹配 brand.csv 中 target_customer 描述的 ICP |
| 9-10 | 多个详细场景，每个场景都有数据支撑和可操作的步骤 |

**检查方法**：
- 搜索内容中的场景描述（"当...时"、"如果你..."、"在 XX 场景下"）
- 检查场景是否匹配目标用户画像（brand.csv target_customer）
- 检查场景中是否有数据/案例支撑

**课程对应**：课程 2-2（TSV 时效性·场景化·可验证）

#### 维度 3：结构清晰（Structure Clarity）—— 权重 15%

**问题**：内容结构是否清晰，AI 能否轻松提取和引用？

| 分数 | 标准 |
|------|------|
| 1-3 | 结构混乱，无明显层次 |
| 4-6 | 有标题但层级不一致，部分内容无小标题 |
| 7-8 | 清晰的 H2/H3 层级，每个 CHUNK 自包含，逻辑流畅 |
| 9-10 | 每个段落都遵循"直接回答 → 展开说明 → 数据支撑"的三段式 |

**检查方法**：
- 复用 audit_content.py 的 CHUNK 检查逻辑
- 检查 H2 标题的层次感和语义连贯性
- 检查每个段落是否以直接回答开头（不铺垫）

**课程对应**：课程 4-1（AI 容易采纳的内容标准——结论先行、条目清晰）

#### 维度 4：关键词覆盖（Keyword Coverage）—— 权重 15%

**问题**：内容是否覆盖了目标关键词及其变体、相关术语？

| 分数 | 标准 |
|------|------|
| 1-3 | 只出现主关键词 1-2 次 |
| 4-6 | 主关键词出现多次，有 1-2 个变体 |
| 7-8 | 主关键词 + 3-5 个变体 + 相关术语 |
| 9-10 | 主关键词 + 变体 + 长尾词 + questions.csv 中的问题文本都在内容中出现 |

**检查方法**：
- 提取 keywords.csv 中与该内容相关的关键词
- 在内容中搜索这些关键词的出现频率
- 检查是否覆盖了同义词、近义词、用户口语化表达

**课程对应**：课程 3-2（关键词三大体系——产品词/痛点词/决策词）

#### 维度 5：可验证性（Verifiability）—— 权重 20%

**问题**：内容中的所有数据和声明是否可验证？

| 分数 | 标准 |
|------|------|
| 1-3 | 没有任何数据或来源 |
| 4-6 | 有少量数据，但来源不明确 |
| 7-8 | 大部分数据有来源标记，来源是权威机构 |
| 9-10 | 所有数据都通过 /web-access 验证，evidence.csv 有对应记录，来源是一手权威来源 |

**检查方法**：
- 检查内容的 DATA_VERIFICATION_LOG 区块
- 检查 evidence.csv 中是否有关联记录
- 检查来源是否在权威白名单内（Baymard、Google、Ahrefs 等）

**课程对应**：课程 2-2（TSV 中的 Verifiable）、课程 2-3（信息增量）

#### 维度 6：语言自然（Language Naturalness）—— 权重 10%

**问题**：内容读起来像专家写的，还是 AI 生成的？

| 分数 | 标准 |
|------|------|
| 1-3 | 明显的 AI 模式：空洞开头（"当然"、"总之"）、过度模板化 |
| 4-6 | 部分 AI 模式但整体可读 |
| 7-8 | 语言自然，无明显 AI slop |
| 9-10 | 像行业专家写的，句式多样，过渡自然 |

**检查方法**：
- 复用 audit_content.py 的 AI_SLOP_PATTERNS 检测
- 检查句式多样性（是否都是"主语+谓语+宾语"的相同结构）
- 检查过渡是否自然（避免"首先...其次...最后...总结"的固定模板）

**课程对应**：课程 2-4（专业术语加通俗表达）、课程 4-1（禁止 AI Slop）

### Step 3：计算加权总分

```
加权分 = 意图覆盖 × 0.25 + 场景匹配 × 0.15 + 结构清晰 × 0.15
       + 关键词覆盖 × 0.15 + 可验证性 × 0.20 + 语言自然 × 0.10
```

**结果分支**：

| 加权分 | 结果 | 下一步 |
|--------|------|--------|
| >= 7.0 | **PASS** | 内容可以发布 |
| 5.0-6.9 | **NEEDS_REVISION** | 修复薄弱维度后重新评分 |
| < 5.0 | **FAIL** | 需要重大修改，不建议发布 |

### Step 4：保存评分记录

调用脚本保存：
```bash
cd $(git rev-parse --show-toplevel)
python 07-prepublish/scripts/score_quality.py \
  --file "<内容文件路径>" \
  --content-id "<ct_xxx>" \
  --scores '{"intent_coverage":8,"scene_matching":7,...}'
```

## 注意事项
- 预发布评分应在格式审计通过后再执行
- 每个维度的评分必须有理由支撑，不能只给分数
- 低于 7.0 的内容必须给出具体修改建议
- 评分记录保存到 `prepublish_score.csv`，用于报告中的质量趋势分析
